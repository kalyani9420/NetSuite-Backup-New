Celigo -> 
Celigo Platform, data is moving from one point to another. This is called a flow.
A flow can include more than two endpoints. A flow can do more than just move data. It can transform, remove, filter, and process data in other ways. 

Connection ->
A connection is how the Celigo Platform communicates with the application

Integration -> 
Folder containing flows

Endpoint -> 
Application, database, or other data source used in a flow

Flow -> 
Stream of data between two or more systems

Important -> 
The endpoints refer to the application or data source, NOT the data itself. 

Source & Destination ->
At least two endpoints in flow - 
The endpoint providing the data (source) 
The endpoint receiving the data (destination) 

Import & Export ->
Export - A configuration record for data pulled from a source 
Import - A configuration record for data brought into a destination 

Lookup ->
We can also create a lookup. Like an export, a lookup retrieves data from an endpoint.
Unlike an export, the information from a lookup doesn't send a data record into a flow. 
Instead, a lookup merges data to an existing record. 
This data usually provides context or complementary information for the existing record, which is needed for the flow.
Lookup could be done with in same endpoint as well as another endpoint


Components ->
1.connections
2.exports
3.imports
4.lookups
these are standalone resource objects being independent from a flow or integration means these components can be used in multiple flows or integrations.
When reusing a standalone resource object (whether it's a connection, export, import, or lookup), it's still one object.
If you make a change to that object in any of its uses, that change is made everywhere the object is used.
hence, it's recommended to create a new component (connection, export, import, or lookup) for each flow.

Example , 
A flow component is used in two flows. In Flow B, the component was changed. What will happen to the component in Flow A?
Ans.It changed when the component changed in Flow B.

Type of Mapping ->
Mapping is part of the flow builder or data loader process where you will tell integrator.io where data from the source application needs to go when it is imported into the destination application. 

Standard Mapping -> The default mapping type is standard mapping. 
This is used when you have a field from the source that will go into a field in the destination. 
For example, lets say the source has a customer name field and the destination has a Name field, this mapping type will work to move the customer's name from the source application to the destination application.

Hard-Coded Mapping -> Hard-Coded Mapping is for when you have a field in the destination application that will always be the same across all records. 
For a field that will be hard-coded, you will need to enter the value for this field. 
A great example of this is a checkbox, this is a great way to map whether you want that checkbox to be checked or not across all records!

Multi-Field Mapping -> Multi-Field Mapping is when you want to take more than one field from the source application and combine those fields into one field in the destination application, or transform data from the source application using field variables. 
More information on field variables can be found here. 

Lookup Mapping -> Lookup field mapping allows you to use a table or other location to populate fields. 
There are two options when you are setting up a lookup mapping, dynamic and static.

Use when creating a webhook

Four types: Basic, HMAC, Secret URL, and Token


The table below lists an overview of the types : 
Basic-> Uses username and password credentials from an external system -> Authentication and Verification
Cookie -> Places pieces of data into each http request header sent to the application; a session-based method. -> Authentication
Custom -> Allows you to define a unique authentication method not listed as another option. User must configure http request fields, which can be encrypted or unencrypted -> Authentication
Digest -> Calls server to receive data, which it then sends back to the server encrypted with the username and other information -> Authentication
HMAC ->	Allows you to set what type of encryption and encoding needs to be done. Encrypted values sent across with secret keys; most secure -> Verification
OAuth 2.0 -> Gives limited access to HTTP service on behalf of an account -> Authentication
Secret URL -> Creates a custom parameter inside the listener URL to verify the URL is valid; sent only once; least secure -> Verification
Token -> Issues API key and password after verified server request. This is used to communicate with the server and expires after a certain amount of time. Path field to locate this must be specified in the Celigo platform -> Authentication and Verification
WSSE -> Use when a connecting application expecting crypto hash generated out of the username and the password. This value is sent over as part of the header. This type is available for HTTP connectors -> Authentication

Pages ->
The Celigo platform puts exported records into pages. 
These pages are chunks of data, which are more manageable for processing. 
They move through the flow one by one, until the export is complete. 
By default, there are 20 records per page. This page size can be modified in export settings in the Advanced section. 
This doesn't apply to webhook listeners, where data is sent in real-time and often a single record.


Total File Size -> 
While you can adjust the number of records per page, that doesn't affect the total file size allowed.
The limit is 5 MB per page. Fewer records per page doesn't necessarily result in a smaller file size. 
As shown in the examples on the right, total file size is dependent on the amount of data in a record.

Important items to note about dynamic lookups:

A destination value is being mapped to a destination value. The difference is the destination value found is usually from a different destination API than where it's being mapped.
The dynamic lookup settings window is where references to the source can be added for filters/searches.
The source field on a mapping row isn't used in dynamic lookups, so it can be blank. (This is shown in screenshots below.) Some users add a source field on the mapping row to serve as a note to themselves about what the dynamic lookup involves, but it has no function in the dynamic lookup process.
The user-created filter needs to return only one value or the dynamic lookup will fail. This is why fields like ID numbers are usually used in dynamic lookups. Email addresses or names are fields which may match more than one record.

Results Mappings (inside a source for lookup)

Every lookup in a flow requires results mapping. This type of mapping instructs the Celigo platform how to handle the data resulting from a lookup. Results mapping:

carries data retrieved from the lookup into the next step of the flow
is applied at the record level
gives the data a source record field name to use later in the flow
shows five options: 
data - result record of lookup
errors - error from processed record
status code - http status code from processing record
ignored - true/false, indicate wheather record process throuth this step is ignored 
dataURI - link the data which gathered in processing 



Source record field
A source record field name needs to be assigned to hold the lookup result data. This can be an existing field name or a new one.
The data with its assigned record field name will be merged into the source exported data. It can be used later in the flow.
Note: If you assign an existing name which is at the top level of the source data's JSON object, the lookup results data will overwrite that field.


Response Mappings (between source and destination)

Response mapping instructs the Celigo platform how to handle an import/destination application's response to data. While lookups require results mapping, an import/destination does not require response mapping.

Response mapping takes place between two imports/destinations, since the data from response mapping flows into another endpoint. Because of this, it can't be the last step in a flow.


Response mapping:

Is applied at the record level

Shows the following options: 
id - id of data imported
errors - error from processed record
status code - http status code from processing record
ignored - true/false, indicate wheather record process throuth this step is ignored 
dataURI - link the data which gathered in processing 
_json - return json formatted response of import
headers - header value of responece


Source record field - hold data will merged to flow into source record for later use in flow

Flow Branching -
Instead of making seperate flow for each criteria, 
Branching allows data to flow in certain directions based on defined criteria. 
If the records match the criteria, it proceeds through to the next sequential steps of the branch. 
If the data does not match the criteria, it moves to the next branch.

Type of matching in branching : 
1.First matching : First matching flows records sequentially through the branches. Records proceed through the first branch where the record matches the branching criteria. 
And matching will stop once criteria meet in first matched branch , This is the only branch where they proceed.
2.All Matching : All Matching means records will proceed through all branches where the conditions are met, regardless of the sequence in which the branches appear.
Criteria will match to all branches if it match then proceed and if it not matched then criteria will match to other next branches

Flow Schedule : 
Flows using standard exports can be scheduled. Real-time exports, which use webhook listeners, cannot. As covered in an earlier course, those are triggered by the source
The two schedule types are: 
Preset: This is the default option. There are 10 preset options ranging from weekly to every 15 minutes.

Cron expression: This is a customizable option. It can specify month, day of month, day of week, hour, and minute.



After the question mark in the Schedule field, there are five positions:

Minute (0-59)
Hour (0-23)
Day of the month (1-31)
Month (1-12)
Day of the Week (0-6 with 0 as Sunday)
The positions are separated by spaces. An asterisk represents "every".

Minute	Hour	Day of Month	Month	Day of the Week
20	6	1,15		*	*



The minute and hour options also allow you to select "every n minutes/hours". Those usually appear as */n in the Schedule field. Examples: 

*/10 in minutes = every 10 minutes

*/23 in hours = every 23 hours

Note: In schedules where every n minutes is used and n is greater than 10, the format changes. The * in the expression changes to 10-59. 
For example: 10-59/15 = every 15 minutes.

This is to indicate the flow will run every 15 minutes starting at 10 past the hour. (This shift was added so all flows don't start at the same time.)




Schedule Override for Exports:
In addition to a flow schedule, individual exports can also be scheduled. This is helpful when you have multiple exports that need to run on different schedules in the same flow. 
To schedule an export, select the plus (+) sign in the export bubble, and select the Schedule Overrride calendar icon. The Flow schedule override window will appear. This includes the same options that are in the Flow schedule window.

Error Troubleshooting:

Step 1: Identify the Source

There are two general sources of errors:

Internal	
In the error list, these are listed as a step in the flow process.
Examples: Lookup, Transformation, Response Mapping.	

External
In the error list, external errors show the application name.
Examples: Mailchimp, Shopify, ADP.




Step 2: Identify the Error Classification

Next to the Source column in the error list is the Classification column. Based on an error's properties :


Value-
A phone number isn't formatted as expected.
A new value is sent for a status field that can't be changed.


Missing-
A required field is empty or missing.
A resource can't be found at the file path provided.

Connection-
The user doesn't have permission to access or edit a record.
An incorrect username or password was used.

Intermittent-
A connection timed out.
This operation may work when retried.


Rate limit-
Too Many Requests
You exceeded your quota for the requested resource.
This classification was previously labeled "Governance."


Too large-
Your query request was running for too long.
 Response stream exceeded limit of 5242880 bytes.
CANNOT_INSERT_UPDATE_ACTIVATE_ENTITY.
This classification was previously labeled "Governance."


Parse-
A token doesn't conform to the expected syntax of the input language.

Duplicate-
The destination already has an existing record. 
The lookup system finds more than one matching result.

Step 3: Identify the Error Category
Most errors fall into one of three categories: 
connection error,
permissions error, or
data error. 

Step 4: Follow Troubleshooting Steps for the Error Category
With the error category identified, you can proceed using troubleshooting steps for that category. Let's begin by looking at connection errors.
This can be addressed with the following steps:

Troubleshooting Connection Errors - 
Retry the step or restart the flow - 
The error may have been temporary. In the error list, select the checkbox next to the errors to retry or you can retry them all. Open the Retry drop-down box and make your selection. 

Check the connector status in the Celigo Platform - 
Check the connections for each of the endpoints. A red dot next to a connection indicates it's offline. Edit the connection to see if any credentials or other settings need to be adjusted or re-entered.

Check the application's status online - 
Many applications have a status page, where the company can alert users of any problems. If the endpoint you're using has such a page, refer to it to see if there are outstanding issues that may affect the flow. 

Contact the application's support team - 
Some errors are caused by issues outside an application's interaction with the Celigo Platform. Contact the application's support team for assistance. Providing them with the error message and code could help pinpoint the issue.


Troubleshooting Permissions Errors - 
Check the error message
Verify credentials and/or permissions in the Celigo Platform
Review permissions in the endpoint/application's external system


Troubleshooting Data Errors - 
Errors classified as Parse, Value, or Duplicate are often types of data errors, which are the most common type of error. 
Errors classified as Missing may also be data errors. And as mentioned earlier, internal errors, which don't always have a classification, are usually data errors.


Troubleshooting Other Errors - 
checking the application's API documentation
contacting their support center to resolve these errors.

Trace Key - Open error -> error field -> trace key
A trace key is a value that uniquely identifies a record. This can be helpful when managing errors and examining flow data.
The Celigo Platform will automatically choose a field to use as a trace key for a record. For example, the id field is selected as the trace key when exporting a record from NetSuite. 

Note: Trace keys can't exceed 128 characters. If the field selected as the trace key has more characters, the trace key will only include the first 128.

You can override the automatic field selected for the trace key and assign a new value, if needed. In the simulation below, you'll walk-through the steps to change the trace key from the default selection to {{record.LastName}}-{{record.Id}}.

Auto-resolve->

Universal Connector -
don't see a pre-built connector (or application-specific connector) for an endpoint, you may still be able to create an integration with that application using universal connectors
The pre-built connectors in the Celigo Platform are universal connectors with the key settings and configurations already set for a specific application. 
Universal connectors (without any pre-configured settings) provide flexibility beyond these pre-built connectors.

You may need to use a universal connector, instead of a pre-built connector, when: 
a pre-built connector isn’t available for the application you want to integrate or
you want to use a new API feature that hasn’t yet been added to the pre-built connector. 


types of universal connectors:
HTTP-
HyperText Transfer Protocol (HTTP)/HyperText Transfer Protocol Secure (HTTPS) allows you to connect to another server. 
It’s used to provide credentials and make data requests. 

REST API (deprecated)-
A type of HTTP connector that conforms to the design principles of the REpresentational State Transfer (REST) architectural style, 
which is a set of specific rules that are not required for all HTTP connections.
As REST API is a subset of HTTP/HTTPS, it does not have the full flexibility and wider range of settings available for HTTP/HTTPS connections.


FTP-
File Transfer Protocol is a standard communication protocol used for the transfer of files to and from a server. You'll also see versions of this named SFTP and FTPS.

Webhook-
This only work in export, not in import
This universal connector is only available when:
the endpoint application has a webhook and
you're creating a connection to an export.
A webhook connector creates a listener URL and sends information to the URL when an event happens in the source application. 
A webhook listener receives the message and then acts based on the content of the message sent.


AS2-
Applicability Statement 2 is a standardized way to move data securely over the internet.  
A combination of encrypted keys, unique IDs, and receipts is used to ensure security.  
To use an AS2 connection, the two businesses/organizations connecting have to agree on what technical specifications and security methods to use, so each endpoint knows what to expect.
This universal connector is typically used in Electronic Data Interchange (EDI) integrations.


Wrapper-
This is a developer toolkit to build connections. It's used only if:
a prebuilt connector isn’t available and 
the connection requires custom code that can’t be created using another universal connector.
While they're more complex to set up and maintain, wrappers allow flexibility to create custom connections, exports, and imports and are often used when the data from a lookup regularly exceeds 5 MB.



GraphQL-
This connector allows you to retrieve data by writing API queries. (A query is a request for information. For example, when you type a phrase into a search engine, that's a search query. A data request to an API is an API query.)
GraphQL improves the processing speed of these information requests. And the size of the data retrieved is less than that of other alternatives, like REST API calls.
GraphQL-based endpoints include Wayfair, Braintree, Instacart, Magento, and PayPal.

Start Line-

Status code's string value
Example: HTTP/2 200 OK


Status codes indicate if the server request was received, understood, accepted, redirected, or rejected. You're probably familiar with the 404 error, when trying to visit a webpage that couldn't be found. 
The status code's string value is a brief explanation of what the code means. Code 404's string value is "Not Found".

Blank Line-
A blank link separates the headers and body. The blank line indicates all the information describing the data was included above.

Body (Optional)-
This is the data being communicated.
If a server is responding with a record request, the record data would reside in the body.
Not all requests or responses include a body. For example, in a GET request to the server, all the information being asked for is included before the body section of the request.

Path Parameters-
A path parameter points to a specific API resource location. Pointing to an individual record is an example of a path parameter. 
Path parameters are separated with a slash (/) and enclose any variables in curly brackets ({ }).  
Example: If the /v1/customers directory includes different types of records, but you only want to export those with an account ID, you can add the account ID field as a path parameter: /v1/customers/{id}

Example: /v1/customers/{id}/company


Query Parameters-
Query parameters allow you to search for records based on key-value pairs. 
They're added to the end of a URI and start with a question mark (?). Key-value pairs, which are determined by the API, are separated with an ampersand (&).
Example: You an export customers created on a certain date who are partners with a query: /v1/customers?createdOn=06-06-2023&type=partner
Because path parameters point to a location, the parameters must be entered in the exact order for the location. Query parameters can include the key-value pairs in any order. 
So, in the example above, /v1/customers?type=partner&createdOn=06-06-2023 would result in the same exported records.

File transfer protocol (FTP) =>

FTP (Not recommended)-
Requires a username and password for the connection
Uses an unsecure connection, allowing anyone who intercepts the data to read it
Typically uses one of two ports: 21 and 22 
While not recommended, this method is still offered since some applications only use this protocol.


SFTP  (SSH file transfer protocol) (Recommended)-
Requires a username, along with a password or authentication key, for the connection
Does not use separate connections for command and data channels
Uses the Secure Shell (SSH) standard to encrypt communications between systems, making this the most secure version of an FTP
Typically uses one port: 22
Important: The Celigo Platform supports authentication keys in PEM format. It doesn't support authentication keys with a passphrase, so the SSH keys should always be generated without a passphrase.


FTPS (File transfer protocol/SSL) -
Requires a username and password for the connection
Encrypts both the command and data channels using transport layer security (TLS). 
 
Can be used in two modes: 
Explicit (default mode in Celigo Platform) - Asks the server to use TLS before the connection is encrypted
Implicit - Both the client and server always use an encrypted connection

Typically uses one of two ports: 
21 for explicit mode
990 for implicit mode
If an FTP server doesn't have SFTP capabilities, FTPS should be preferred over FTP due to its security capabilities.






In the Application details section, you'll enter the host name. This is the location of the FTP server you'll connect to. It may be a:
URI (Example: ftp.mycompany.com) or 
IP address (Example: 172.2.99.249)






Active mode :
With FTP connections, server send request and then client respond

Passive mode:
With FTP connections, the client computer makes the initial connection request to the server. This is the command channel. In active mode, a server then initiates the data channel connection from a port, usually 20. 
In passive mode, the server doesn't start the data channel connection. Instead, using the command channel, the server tells the client computer to send a data connection request to a random port it opened. 
Then the server (passively) listens for this request.
Passive mode is more secure than active mode, with both connections (command and data) originating from the client computer.




User directory is root->
When setting up your FTP connection, there is a checkbox to indicate whether the user directory is the root directory.
The root directory is usually the top-most directory in a computer folder system. All the other directories branch out from it.



Entry parser -> 
In most cases, this optional field is left blank. 
The entry parser field identifies how to handle FTP responses if:
they're uncommon (not Windows or Linux) or 
the server doesn't reply to a command asking for its operating system. 



Enable PGP cryptographic -> 
Select this checkbox if using PGP encryption. 
PGP encryption increases data security when processing and transferring files. (PGP stands for "Pretty Good Privacy".) 
This encryption requires public and private keys, which the Celigo Platform does NOT generate. You'll use a third-party tool to create PGP keys to encrypt and decrypt files.
enter the following information about the encryption:
PGP public key
PGP private key
Private key passphrase and
Compression algorithm



Require socket reuse->
This option is only available when using an FTPS protocol. 
As mentioned earlier, FTPs have two connections to a server (channel and data). While we've reviewed ports, the endpoints of these connections themselves are called sockets. Sockets are bound to a port number.

SQL stands for "structured query language," a programming language used to communicate with databases. It can retrieve, create, insert, update, and delete database records.

Ex,
SELECT Product FROM ProductList WHERE Size='10'

Export type->
All-
Always exports all data


Delta-
Exports only data modified since the last export
If this export type is used, the SQL query should include the expressions: 
lastExportDateTime or
currentExportDateTime

Limit-
Exports a set number of records
The default value is one. The maximum value allowed is 100.


Once-
Only exports data that hasn't already been exported
To mark a record as exported and prevent it from being exported again, an additional SQL query is needed. Below is an example of a query for this:
UPDATE tableName SET exported=true WHERE id={{data.id}}
In the example above, "exported" is a Boolean field to indicate whether the record was updated; id represents an unique identifier for a record.

How would you like the records imported?

Insert-
Adds a new record into a database


Update-
Changes an existing record in a database


Insert or Update-
Checks to see if there's an existing record to update. 
If there is, the record is updated; otherwise, a new record is made.

Ignore existing records (appears for Insert query type)-
This ignores records that already exist in the database. A new record will override an existing one. This provides protection against duplicates.
Since this requires checking if there is an existing record, there is a slight effect on the processing time.

Ignore missing records (appears for Update query type)-
A record must already exist in order for it to be updated. When it doesn't, an error will occur when updating records. If you expect some records to be missing during an import update and want to avoid error messages about it, you can select "ignore missing records".

Concurrency ID lock template-
When a connection is sending multiple requests to a database at the same time, it has a concurrency greater than one. 
If this is the case, a record may be queued for import multiple times, resulting in duplicates.
This field prevents that. The value entered here, which should be a handlebars template, generates a unique ID for each exported record. 
The Celigo Platform enforces an order, so that no two records with the same ID are submitted for import at the same time.


Data URI template-
This field uses a handlebars expressions template to generate dynamic links to the data imported, where possible. 
These links are helpful when troubleshooting errors. They appear on the error dashboard, so you can target specific data tied to an error.


Batch size-
This controls how many files should be retrieved at a single time.  
Below is a list of factors to note about batch size:

This is not the number of total files that can be retrieved.
A larger batch size decreases the number of requests to the database, but other factors need to be considered when setting this number.
A database can have a hard limit on the number of records that can be submitted in a single batch request. 
Batch sizes can't be greater than the page size.

This API is for developers to further customize flows to fit their business needs. integrator.io uses bearer token authentication.

What is a REST API? 
A REST, or REpresentational State Transfer, based application exposes information in the form of information about its resources. 
With a REST API, the user can take actions on the resources available, such as creating a new resource or changing an existing resource.  



What makes an API RESTful?
In order to be considered a REST API, integrator.io must follow certain constraints that make the API easier to use and easier for a developer to learn how to use the API. 
Most REST APIs, including integrator.io use the JSON format. 



Making a call through a REST API:
When you make a call to a RESTful API, you will need to provide two things in the call to specify what action you are wanting to happen. 
An identifier for the resource you would like to include in the call.  This is a URL and is known as the endpoint. 
The operation you would like the server to perform, in the form of an HTTP method. integrator.io supports the following HTTP methods: GET, POST, PUT, DELETE



Creating a Token in integrator.io ->
Resources ->API Token
Authorization type -> Bearer Token


When to Use the API -> 
New Functionality - 
You will want to use the API if a new function is available in integrator.io but the UI (User Interface) has not been updated to support this function yet. 
Static Lookups - 
If you have a large list of items for your static lookup table, it is easier to update or insert into this table using an API request instead of typing it in manually using the UI. 



Making an API Request ->

GET - 
This call retrieves resources where your token has access. 


POST -
This call creates new records or items in the resource where your token has access.   


PUT -
This call updates existing records or items in the resource where your token has access 



DELETE - 
This call deletes resources where your token has access.

MY API ->
My API allows users to build their own API on top of integrator.io. A client application can make a request to a URL created through My API. My API will then execute scripts to return a formatted and authenticated response acceptable to the system.
Because My API makes calls outside of jobs within integrator.io, the logs, errors, and dashboard information will not be updated when data is integrated. There will be no opportunity to retry records that failed to sync within integrator.io like with normal flows. 



REST API vs My API

REST API-
Used to retrieve data, for example, inventory levels from a system like NetSuite for web stores.
Data is always returned in JSON format.
Authentication occurs through an API token created within integrator.io. 
My API-
Utilizes JavaScript to handle authentication or data format issues that can sometimes arise when using REST APIs.
Data can be returned in the response to the original request in the proper format.

Webhook vs My API

Webhooks-
Allow users to expose a URL that data can be pushed to in real-time when a given event occurs in an application.
Run asynchronously. Responses advise of successful data transfers or authentication or data parsing errors.
Cannot return data in response to the original system. For example, an internal Id of the record created in a corresponding import.
My API-
Allows subsequent calls to be made through scripting that can return more information within the response.
